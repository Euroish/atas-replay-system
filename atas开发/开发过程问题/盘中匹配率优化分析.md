## 2. 为什么盘中只有 80% 多

核心不是集合竞价，集合竞价可以先放。盘中主要卡在 4 个地方。

### 第一，同毫秒内事件顺序不确定

现在统一规则是：

```
order/cancel → trade → quote
```

但是行情快照可能是：

```
quote → order/cancel → trade
```

也可能是：

```
order → quote → trade
```

也可能一秒内 quote 是撮合系统某个采样点，不是逐笔流最后状态。

你之前已经做过“同秒 quote 放最后”的实验，结果总 mismatch 反而变成 210944，更差，所以不能用一个全局排序规则解决。项目记录里也写了这个实验未采用。[](https://github.com/Euroish/atas-replay-system/blob/main/findings.md)

[findings](https://github.com/Euroish/atas-replay-system/blob/main/findings.md)

结论：

> 盘中必须做“局部排序搜索”，不是固定排序。

---

### 第二，raw book 一旦错，会持续漂移

当前 engine 没有 checkpoint。它从头开始一路累积订单簿。

如果 09:42:10 某个撤单没有正确扣掉，那么 09:42:10 之后，这个价位的量会一直多出来，甚至挤占 top10 层级。

所以一个错误不是只影响一个 quote，而是会污染后面很多 quote。

---

### 第三，缺失订单主要集中在盘中高活跃股票

项目记录中 residual diagnostics 显示，top missing-order group 是：

```
missing_trade_order, continuous_am, 300408.SZ: 16744
missing_trade_order, continuous_am, 000711.SZ: 14013
missing_trade_order, continuous_pm, 300408.SZ: 9773
```

也就是说盘中问题集中在连续竞价、高活跃品种、成交相关缺失。[](https://github.com/Euroish/atas-replay-system/blob/main/findings.md)

[findings](https://github.com/Euroish/atas-replay-system/blob/main/findings.md)

这类问题不能只靠修正 order_id 逻辑解决，还需要 quote anchor。

---

### 第四，当前没有“可见盘口层”

`task_plan.md` 明确写了：当前阶段没有做 replay clock、checkpointing、quote-overwrite anchoring。[](https://github.com/Euroish/atas-replay-system/blob/main/task_plan.md)

[task_plan](https://github.com/Euroish/atas-replay-system/blob/main/task_plan.md)

也就是说：

这才符合你要做的 ATAS 类回放。

## 4. 推荐算法：盘中 95%+ 匹配率方案

### 方案 A：快照锚定 Replay，最稳，优先做

每次遇到 quote：

```
1. 先用 raw events 推进 RawBook
2. 对比 RawBook top10 和 quote top10
3. 如果 mismatch 小，保留 raw
4. 如果 mismatch 超过阈值，用 quote top10 重建 VisibleBook
5. 记录 correction，不污染 RawBook 的订单 ID 结构
```

关键点：

```
RawBook 不覆盖
VisibleBook 覆盖
```

因为 quote 只有价格和数量，没有真实 order_id。如果你直接覆盖 RawBook，会制造假订单号，后面成交/撤单又会乱。

应该新增：

```Python
visible_book_engine.py
```

核心结构：

```Python
class VisibleBookEngine:
    def __init__(self):
        self.bid_levels = {}
        self.ask_levels = {}
        self.corrections = []

    def anchor_from_quote(self, quote_row):
        self.ask_levels = quote ask1-ask10
        self.bid_levels = quote bid1-bid10

    def apply_delta_from_events(self, events_between_quotes):
        # 可选：用逐笔事件微调可见盘口
        pass
```

这样盘中 quote 点本身可以接近 100% 匹配；两个 quote 之间用逐笔成交/委托/撤单补动画。

这个方案最适合你要的：

```
heatmap 精准
DOM 精准
回放过程中盘口不飘
匹配率 95%+
```

---

### 方案 B：同毫秒局部重排，作为 raw 优化

不要用全局排序，而是对每个 quote 附近的小窗口做搜索。

例如对每个 quote：

```
窗口 = [quote.ts_ms - 200ms, quote.ts_ms + 200ms]
```

候选顺序：

```
1. order/cancel → trade → quote
2. trade → order/cancel → quote
3. quote → order/cancel → trade
4. order → quote → cancel/trade
5. 按 source_seq 原始顺序
```

对每种顺序模拟一次，选择 mismatch 最小的顺序。

伪代码：

```Python
for quote in quotes:
    candidates = build_ordering_candidates(events_near_quote)
    best_state = None
    best_score = inf

    for candidate in candidates:
        test_engine = engine.clone()
        test_engine.apply_events(candidate.before_quote)
        score = compare_top10(test_engine, quote)

        if score < best_score:
            best_score = score
            best_state = test_engine

    engine = best_state
```

但注意：

> 这个只能提高 raw 匹配率，不能保证 95%+。因为有些数据本身不是完整 order-by-order book feed。

所以它是第二优先级，不是核心。

---

### 方案 C：残差修正，不改变订单 ID

当前最大风险是直接补假订单。不能这么做。

正确做法是给 VisibleBook 加一个 correction layer：

```
raw_level_qty + correction_qty = visible_level_qty
```

例如 quote 显示：

```
bid1 10.00 5000
```

raw book 显示：

```
bid1 10.00 4200
```

则：

```
correction_qty = +800
```

后续如果 raw event 继续变化：

```
visible_qty = raw_qty + correction_qty
```

到下一个 quote 再重新校准。

这样不会破坏原始订单簿，也能保证显示盘口跟真实快照一致。

这样视觉上不会突然跳变。


## 6. 最关键结论

你现在的系统已经完成了“可测量 raw 偏差”的阶段，但还不是“高精度盘口复现”的阶段。

当前 raw 匹配率 80% 多的根本原因是：

```
逐笔事件流 ≠ 十档快照的严格前置事件集合
```

所以继续只修 `OrderBookEngine`，收益会越来越小。

最优路线是：

```
RawBook 继续保留，负责真实订单流和 footprint；

```

目标应该改成：

```

raw_match_rate 作为诊断指标，不强行追 95%
```

如果你强行要求 **raw_match_rate 本身 95%+**，那需要数据源本身提供完整、严格排序、无缺失的 order-by-order 全量订单簿事件；以当前 CSV 结构和现有残差表现看，不现实。
